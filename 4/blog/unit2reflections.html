Rationale

This unit necessitated creating HTML which described the structure of the content of my website according to the design documentation that I created in Unit 1.

The scenario, "Diya's recon," is satisfied to a basic degree by the use of accessibility features of HTML to describe content that is otherwise not accessible. This includes captions of images and other information which is suggested for accessibility reasons on the MDN HTML Reference.

The file /blog/linux-recommendations.html meets the needs of the scenario titled, "Yanjo needs Linux recommendations." The file does this by including the tags and difficulty rating design elements, which were introduced into the design to meet the needs of Yanjo.

The file /blog/index.html meets the needs of the scenario titled "Udonavon needs Emacs documentation." The file does this by summarizing the content of the blog post, which meets the needs of an "at a glance", executive summary or "TL;DR" for busy people.

Not every scenario has been fulfilled at this point in the implementation of the website. There is an inordinate amount of time that has been spent on this unit, so to progress through the course some of the scenarios and design elements will be implemented later.

Critique of the sample HTML documents

Sample 1

Some of the issues with the first sample document are:

  • a lack of indentation of child elements of the head and body tags
  • the head element doesn't not contain the right information, the head should minimally contain:
    • metadata (using a meta tag element) for the character set the document is encoded in, and
    • a title element with text content which actually describes the document accurately
  • the html, document root element should have a lang attribute to indicate what language the majority of the text content is displayed in, or the langauage the website (overall) is provided in
  • inconsistency in the amount of whitespace following the opening tag before the beginning of the text content (some elements and tags begin with one space character, others do not)
  • alike the last list item, some elements may or may not have consistency in the amount of whitespace before the closing tag (this is difficult to ascertain because of the horrendous nature and lack of correct syntax in the table
  • inconsistency in the capitalization (or case) of html tags
  • incomplete elements, with some elements not being closed at all (e.g., the paragraph element immediately following the first H1 header is unclosed)
  • improperly nested html elements, with the beginnings of some elements within the space of other elements, that is (intersecting nodes, which are not allowed)
  • very long lines, which extend beyond eighty characters, but also inconsistency in that as well (see lines 9 and 10 of the document)
  • use of deprecated tags
  • no document type declaration or namespace declaration
  • No code commentary or useful whitespace to separeate logically or semantically different areas of the code or content
  • Poor style of attribute values (see the src attribute of the image element) and inconsistency thereof (its okay for boolean values, or ordinal values less than ten to not be quoted, but all other attribute values should, ideally, be quoted to ensure they are properly terminated when parsed)
  • use of obsolete attributes (border in table)
There are, however, other mistakes in the document. These other mistakes are opinionated, and not necessarily wrong.

I improved sample1.html by giving it a title, specifying the written language of the document and the encoding of the HTML document. I also used semantic tags to section the different parts of the document (article and section semantic elements partition the document neatly, legibly, and meaningfully).

I included the header/navigational content of my website to return users to my actual website from this blog post.

I fixed grammar, spelling, indentation, line length, and more.

I fixed the lists, the table, and the nesting (the issue with intersecting nodes).

Sample 2

The second sample has only one problem according to the transitional document type declaration for XHTML: there is a complete lack of content, which would confuse many users.

You could argue that the title element is also incorrect, as an empty document is not necessarily a document that should have a title, therefore titling the document "Untitled Document" is misleading because of the implication of existing content.

I improved sample2.html by changing the document to a modern HTML5 document type, and adding a little bit of content: an under construction notification. I also added indentation and extra new lines to separate parts of the HTML source code. I also corrected the title by making it summarize the entire webpage, "Page under construction."

The reference for label elements is the MDN HTML reference.

The reference for HTML4 is W3.

Learning outcomes

I believe I have met the learning outcomes of this unit, because I have learned to and did write several readable, semantic HTML5 web pages for my website. I can maintain this code easily, and I am able to easily style it differently with different classless CSS files, showing that the HTML is not deprecated.

The full learning outcome mappings are written in the document which is attached to the blog post on AthabascaU's Landing.

Successes and difficulties

I have been a computer geek my whole life, so gaining a deeper understanding of HTML was not difficult. What was difficult was returning to the work on this unit several times, and each time being struck with the task of keeping the idea and structure of the website in my head. The design documentation was sat next to me on my desk top, but I hadn't reviewed it and so assessing what was needed was sometimes a little difficult.

The other success I had with this unit was establishing some unwritten rules for how I want the site to be implemented, ie the semantic structure of the website.

Hindsight

Were I to repeat this unit's effort, I would first take a hard, long look at other students work to see what they did in regard to reflecting on the process. Learning HTML and structuring content is not difficult for me, but in the previous unit and this unit I have struggled to map my activities to learning outcomes or to reflect on the learning process and create a blog post for the course administrator to review and grade.

The other thing I would do differently is create a template for my page, which has the head, body, header with nav, and footer elements filled in. I'd write a function to update the copyright and modification time information in Emacs Lisp, or learn the appropriate functions that indeed already exist in Emacs, and I would let Emacs format the file for me rather than using my own ad-hoc formatting. Howevever, I think I have still managed to do well in this unit.
